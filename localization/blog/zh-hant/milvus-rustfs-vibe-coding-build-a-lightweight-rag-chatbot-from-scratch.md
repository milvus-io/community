<p><em>本部落格由 Milvus 社群貢獻者馬景和撰寫</em> <em>，並經允許在此發表。</em></p>
<p>我想要一個聊天機器人，可以回答我自己文件中的問題，而且我想要完全控制它背後的堆疊 - 從物件儲存到聊天介面。這促使我以<a href="https://milvus.io/">Milvus</a>和<a href="https://rustfs.com/">RustFS</a>為核心，建立了一個輕量級的 RAG 聊天機。</p>
<p><a href="https://milvus.io/">Milvus</a>是建立 RAG 應用程式最廣泛採用的開放原始碼向量資料庫。它將運算與儲存分開，將熱資料保存在記憶體或 SSD 上以進行快速搜尋，同時依賴底下的物件儲存以進行可擴充、具成本效益的資料管理。由於它可與 S3 相容的儲存一起運作，因此自然適合這個專案。</p>
<p>對於儲存層，我選擇了<a href="https://rustfs.com/">RustFS</a>，這是用 Rust 寫成的開放原始碼 S3 相容物件儲存系統。它可以透過二進位、Docker 或 Helm 圖表部署。雖然它仍處於 alpha 狀態，不建議用於生產工作負載，但對於這次的建置而言，它已經夠穩定了。</p>
<p>基礎架構就位後，我需要一個知識庫來查詢。RustFS 文件 - 約 80 個 Markdown 檔案 - 是一個方便的起點。我將文件分塊、產生嵌入、儲存在 Milvus 中，剩下的就用 vibe 編碼：後端使用<a href="https://fastapi.tiangolo.com/">FastAPI</a>，聊天介面使用<a href="https://nextjs.org/">Next.js</a>。</p>
<p>在這篇文章中，我將介紹端對端的完整系統。程式碼可在 https://github.com/majinghe/chatbot 取得。這是一個可運作的原型，而不是一個可供生產的系統，但我們的目標是提供一個清晰、可擴充的建置，讓您可以為自己的用途進行調整。以下每節都會介紹一個層次，從基礎架構到前端。</p>

<p>Milvus 可以使用任何 S3 相容的物件儲存，雖然 MinIO 是標準設定中的預設後端。由於 MinIO 不再接受社群貢獻，因此在本範例中我們將以 RustFS 取代它。</p>
<p>要進行這項變更，請更新 Milvus 套件庫內 configs/milvus.yaml 中的物件儲存設定。相關章節如下：</p>

<p>有兩種方式可以套用此變更：</p>
<ul>
<li><strong>掛載本機設定檔案。</strong>複製 configs/milvus.yaml 到本機，更新 MinIO 欄位以指向 RustFS，然後透過 Docker volume 將其掛載到容器中。</li>
<li><strong>在啟動時使用</strong> <strong>yq****</strong><strong>進行修補</strong> <strong>。</strong>修改容器的指令，在 Milvus 程序啟動前，針對 /milvus/configs/milvus.yaml 執行 yq。</li>
</ul>

<h3 id="Reading-and-Chunking-the-Docs" class="common-anchor-header">閱讀並將文件分塊</h3><p>腳本會遞迴讀取 docs 資料夾中的每個 .md 檔案，然後按新行將每個檔案的內容分割成小塊：</p>
<pre><code translate="no"><span class="hljs-comment"># 3. Read Markdown files</span>
<span class="hljs-keyword">def</span> <span class="hljs-title function_">load_markdown_files</span>(<span class="hljs-params">folder</span>):
    files = glob.glob(os.path.join(folder, <span class="hljs-string">&quot;**&quot;</span>, <span class="hljs-string">&quot;*.md&quot;</span>), recursive=<span class="hljs-literal">True</span>)
    docs = []
    <span class="hljs-keyword">for</span> f <span class="hljs-keyword">in</span> files:
        <span class="hljs-keyword">with</span> <span class="hljs-built_in">open</span>(f, <span class="hljs-string">&quot;r&quot;</span>, encoding=<span class="hljs-string">&quot;utf-8&quot;</span>) <span class="hljs-keyword">as</span> fp:
            docs.append(fp.read())
    <span class="hljs-keyword">return</span> docs

<span class="hljs-comment"># 4. Split documents (simple paragraph-based splitting)</span>
<span class="hljs-keyword">def</span> <span class="hljs-title function_">split_into_chunks</span>(<span class="hljs-params">text, max_len=<span class="hljs-number">500</span></span>):
    chunks, current = [], []
    <span class="hljs-keyword">for</span> line <span class="hljs-keyword">in</span> text.split(<span class="hljs-string">&quot;\n&quot;</span>):
        <span class="hljs-keyword">if</span> <span class="hljs-built_in">len</span>(<span class="hljs-string">&quot; &quot;</span>.join(current)) + <span class="hljs-built_in">len</span>(line) &lt; max_len:
            current.append(line)
        <span class="hljs-keyword">else</span>:
            chunks.append(<span class="hljs-string">&quot; &quot;</span>.join(current))
            current = [line]
    <span class="hljs-keyword">if</span> current:
        chunks.append(<span class="hljs-string">&quot; &quot;</span>.join(current))
    <span class="hljs-keyword">return</span> chunks

<ul>
<li><strong><a href="http://milvus.io">Milvus</a></strong> <strong>+</strong> <strong><a href="https://rustfs.com/">RustFS</a></strong> <strong>與 Docker Compose。</strong>Milvus 以獨立模式執行，並以 RustFS 作為其物件儲存後端，取代預設的 MinIO。共有四種服務：etcd、Milvus、RustFS 和 Attu。</li>
<li><strong>將知識庫矢量化。</strong>RustFS 的說明文件 - 80 個 Markdown 檔案 - 經過分塊、text-embedding-3-large 的嵌入，並儲存在 Milvus 中，成為 466 個向量。</li>
<li><strong>RAG 管道。</strong>在查詢時，使用者的問題會以同樣的方式嵌入，Milvus 會擷取語意最相似的三個區塊，而 GPT-5 會根據這些文件產生答案。</li>
<li><strong>聊天機使用者介面。</strong>FastAPI 將管道包裝在單一 POST 端點中；Next.js 則在管道前方放置聊天視窗。再也不用跳到終端機來發問了。</li>
</ul>
<p>我從過程中得到的一些啟發：</p>
<ul>
<li><strong><a href="https://milvus.io/docs">Milvus 的說明文件</a></strong> <strong>很棒。</strong>尤其是部署部分 - 清晰、完整、易於遵循。</li>
<li><strong><a href="https://rustfs.com/">RustFS</a></strong> <strong>作為物件儲存的後端非常好用。</strong>把它加入 MinIO 所花的時間比預期的還要少。</li>
<li><strong>Vibe 的編碼速度很快，直到範圍接管為止。</strong>從 Milvus 到 RAG 到聊天機器人，再到「也許我該把整件事 Docker 化」。需求不會自行匯合。</li>
<li><strong>調試比閱讀教得更多。</strong>這次建置中的每一次失敗，都讓下一節的點選速度比任何文件都要快。</li>
</ul>
