<p><a href="https://github.com/google-research/google-research/tree/master/scann">ScaNN</a> - это ответ Google на привычную проблему крупномасштабного векторного поиска: <strong>как увеличить пропускную способность запросов и снизить потребление памяти без неприемлемого ущерба для качества результатов.</strong> Концептуально ScaNN основывается на классическом рецепте IVF+PQ - грубая кластеризация плюс агрессивное квантование по продуктам - но включает два важных новшества, которые существенно сдвигают границу производительности:</p>
<ul>
<li><p><strong>Задача квантования с учетом баллов</strong>, которая лучше сохраняет относительное упорядочивание истинных соседей, улучшая качество ранжирования даже при сильном сжатии.</p></li>
<li><p><strong>FastScan</strong> - это оптимизированный для SIMD 4-битный путь поиска PQ, который уменьшает традиционное узкое место в памяти, сохраняя больше работы в регистрах процессора.</p></li>
</ul>
<p>На практике он является сильным выбором, когда вы готовы пожертвовать некоторым количеством отзыва ради высокого QPS и гораздо меньшего объема памяти (часто сжимая векторы до ~1/16 от исходного размера), например, в рекомендательных нагрузках, не чувствительных к отзыву.</p>
<p>В этом посте мы вновь рассмотрим IVFPQ в качестве базового уровня, изучим ключевые оптимизации, которые ScaNN вводит поверх него, и завершим обсуждение экспериментальными результатами, основанными на измеренной производительности.</p>

<p>Прежде чем представить ScaNN, мы кратко расскажем о IVFPQ, поскольку ScaNN построен на основе того же общего фреймворка.</p>
<p><strong>IVFPQ расшифровывается как Inverted File with Product Quantization</strong>- алгоритм, используемый для эффективного и масштабного поиска ближайших соседей (ANN) в высокоразмерных векторных базах данных. Это гибридный подход, сочетающий две техники - <strong>инвертированный индекс файла (IVF)</strong> и <strong>квантование продукта (PQ)</strong>- для баланса скорости поиска, использования памяти и точности.</p>
<h3 id="How-IVFPQ-Works" class="common-anchor-header">Принцип работы IVFPQ</h3><p>Процесс включает в себя два основных этапа индексирования и поиска:</p>
<ul>
<li>Уровень ЭКО: векторы кластеризуются в инвертированные списки (кластеры) <code translate="no">nlist</code>. Во время запроса посещается только подмножество кластеров (<code translate="no">nprobe</code>), чтобы найти компромисс между отзывом и задержкой/пропускной способностью.</li>
</ul>
<p>
  <span class="img-wrapper">
    <img translate="no" src="https://assets.zilliz.com/ivf1_5e3d29c392.png" alt="" class="doc-image" id="" />
    <span></span>
  </span>
</p>
<ul>
<li>Уровень PQ: внутри каждого посещенного кластера каждый D-мерный вектор разбивается на m подвекторов, каждый из которых имеет размерность (D/m). Каждый субвектор квантуется путем присвоения ему ближайшего центроида в его субкодексе. Если в подкодексе 256 центроидов, то каждый подвектор может быть представлен кодом <code translate="no">uint8</code> (идентификатор в диапазоне [0, 255]).</li>
</ul>
<p>
  <span class="img-wrapper">
    <img translate="no" src="https://assets.zilliz.com/pq2_6695c9cc6f.png" alt="" class="doc-image" id="" />
    <span></span>
  </span>
</p>
<p>Вычисление расстояния может быть переписано как сумма по подвекторам:</p>
<p><strong>D(q, X) = D(q, u0) + D(q, u1) + D(q, u2) + ... + D(q, un)</strong></p>
<p><strong>= L(q, id1) + L(q, id2) + L(q, id3) + ... + L(q, idn)</strong></p>
<p>Здесь L представляет собой таблицу поиска. Во время запроса строится таблица поиска, в которой записывается расстояние между запросом и каждым квантованным подвектором. Все последующие вычисления расстояний преобразуются в поиск по таблице с последующим суммированием.</p>
<p>
  <span class="img-wrapper">
    <img translate="no" src="https://assets.zilliz.com/query_vector3_75d47bdd53.png" alt="" class="doc-image" id="" />
    <span></span>
  </span>
</p>
<p>Например, для 128-мерных векторов, разбитых на 32 субвектора по 4 измерения каждый, если каждый субвектор кодируется идентификатором <code translate="no">uint8</code>, стоимость хранения одного вектора снижается с (128 x 4) байт до (32 x 1) байт - сокращение на 1/16.</p>

<ol>
<li><p>Квантование: ScaNN предлагает цель, выходящую за рамки простой замены каждого субвектора на ближайший к нему центроид k-средних (т. е. минимизация ошибки реконструкции).</p></li>
<li><p>Эффективность поиска: ScaNN ускоряет поиск на основе LUT, который часто ограничен памятью, с помощью SIMD-дружественного пути FastScan.</p></li>
</ol>

<p>Почему к параллельной компоненте должен применяться больший штраф?</p>
<p>Во-первых, если x является ближайшим соседом q1, то направление x аналогично q1, поэтому параллельную компоненту x можно приблизительно считать параллельной q1, и эта параллельная компонента увеличит ошибку.</p>

<h3 id="4-bit-PQ-FastScan" class="common-anchor-header">4-битный PQ FastScan</h3><p>Сначала рассмотрим процесс вычисления PQ: во время запроса расстояния между центрами кластеров запроса и субвектора предварительно вычисляются для построения таблицы поиска. Затем вычисление расстояний производится путем обращения к таблице для получения расстояний между сегментами и их суммирования.</p>
<p>
  <span class="img-wrapper">
    <img translate="no" src="https://assets.zilliz.com/PQ_Fast_Scan1_248c53bc93.png" alt="" class="doc-image" id="" />
    <span></span>
  </span>
</p>
<p>Однако частые обращения к памяти все равно становятся узким местом в производительности. Если таблицу поиска сделать достаточно маленькой, чтобы она помещалась в регистры, операции чтения памяти можно преобразовать в эффективные SIMD-инструкции процессора.</p>
<p>Сначала каждый субвектор разбивается на 16 классов, так что 4-битное значение может представлять центр кластера - отсюда и название "4-битный PQ". Затем расстояния, обычно представляемые в виде плавающих чисел, преобразуются в uint8 с помощью скалярной квантизации (SQ). Таким образом, таблица поиска для одного субвектора может храниться в регистре, используя 16 × 8 = 128 бит.</p>
<p>
  <span class="img-wrapper">
    <img translate="no" src="https://assets.zilliz.com/PQ_Fast_Scan2_07b9589195.png" alt="" class="doc-image" id="" />
    <span></span>
  </span>
</p>
<p>Наконец, рассмотрим схему хранения регистров (на примере набора инструкций AVX2): субвекторы 32 векторов помещаются в 128-битный регистр, объединенный с таблицей поиска. Операция "поиск" может быть эффективно выполнена с помощью одной SIMD-инструкции процессора.</p>

    </button></h2><p>Используя бенчмарк векторных баз данных <a href="https://github.com/zilliztech/VectorDBBench">VectorDBBench</a>, мы провели простой тест. Преимущество производительности ScaNN над традиционными IVFFLAT и IVF_PQ вполне очевидно. После интеграции в Milvus на наборе данных Cohere1M при одинаковом коэффициенте отзыва QPS достигает 5 раз по сравнению с IVFFLAT и 6 раз по сравнению с IVF_PQ.</p>
<p>Однако QPS немного ниже, чем у индексов на основе графов, таких как HNSW, поэтому он не является первым выбором для сценариев с высоким коэффициентом запоминания. Но для сценариев с меньшим количеством отзывов это приемлемо (например, в некоторых рекомендательных системах), использование ScaNN без загрузки исходных данных позволяет достичь впечатляющего QPS при крайне низком объеме памяти (1/16 от исходных данных), что делает его отличным выбором индекса.</p>

Согласовывая цель квантования с качеством ранжирования и устраняя узкие места памяти во внутреннем цикле, ScaNN сочетает квантование с учетом баллов с 4-битным PQ FastScan-путем, который превращает традиционно ограниченный памятью процесс в эффективное, SIMD-дружественное вычисление.</p>
<p>На практике это дает ScaNN четкую и хорошо определенную нишу. Он не предназначен для замены индексов на основе графов, таких как HNSW, в условиях большого количества обращений. Напротив, для нечувствительных к отзывам рабочих нагрузок с ограниченным бюджетом памяти ScaNN обеспечивает высокую пропускную способность при очень малой занимаемой площади. В наших экспериментах, после интеграции в Milvus VectorDB, ScaNN достигла примерно 5× QPS по сравнению с IVFFLAT на наборе данных Cohere1M, что делает ее отличным выбором для высокопроизводительного поиска ANN с низкой задержкой, где сжатие и эффективность важнее идеального запоминания.</p>
